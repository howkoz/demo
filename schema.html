Portal 9094 — Backend Architecture & Schema

This is the production backend design the demo at port 9094 is built against. The demo itself is 100% synthetic — every widget, filter, and page maps 1:1 to a real view/table defined below. Nothing in the UI would need to move if the JSON files were swapped for live API responses tomorrow.

1. System topology

+-----------------------------+ | Browser (port 9094) | | static HTML/JS/CSS | +--------------+--------------+ | HTTPS (REST) | +--------------v--------------+ | Portal 9094 API | | Flask / FastAPI | | read-only, cache-friendly | +--------------+--------------+ | +--------------+--------------+ | | +---------v---------+ +---------v---------+ | DW (MSSQL) | | UnityDBStage | | orders, dist, | | analytic store: | | txn detail, | | gateway recon, | | bonus/enroll | | providers, events,| | | | fraud, markets | +---------+---------+ +---------+---------+ ^ ^ | | +---------+---------+ +---------+---------+ | CDC / nightly | | ETL: gateway API | | dimension sync | | pullers (Python) | +-------------------+ +-------------------+

Two data planes, read-only API in front.

  • DW (MSSQL) is the source of truth for orders, distributors, transactions, commissions, enrollment.
  • UnityDBStage is the unified analytic store: per-gateway reconciliation tables, cross-gateway views, provider directory, activity events, fraud cases, and market metadata all live here.

The Portal 9094 API is a thin Flask/FastAPI read-only layer. No writes from the browser. All caching is keyed on (endpoint, market, days) with a 15-minute TTL.

4.4 Views the API hits

-- v_approval_daily: powers ticker, approval trend, chargeback health.
CREATE OR REPLACE VIEW v_approval_daily AS
SELECT
  action_date AS date,
  country     AS market,
  COUNT(*) AS orders,
  SUM(CASE WHEN result IN ('approved','settled','captured') THEN 1 ELSE 0 END) AS approved,
  SUM(CASE WHEN result IN ('declined','failed','error') THEN 1 ELSE 0 END)     AS declined,
  SUM(CASE WHEN result IN ('approved','settled','captured') THEN amount_usd ELSE 0 END) AS volume_usd,
  SUM(CASE WHEN result IN ('chargeback','dispute') THEN amount_usd ELSE 0 END) AS chargebacks,
  SUM(CASE WHEN result = 'refunded' THEN amount_usd ELSE 0 END) AS refunds
FROM v_all_gateway_actions
WHERE action_date IS NOT NULL
GROUP BY 1, 2;
-- v_statement_vs_actions_monthly: powers recon gauge.
CREATE OR REPLACE VIEW v_statement_vs_actions_monthly AS
SELECT
  gateway,
  strftime(action_date, '%Y-%m') AS month,
  COUNT(a.gateway_txn_id)        AS gateway_rows,
  SUM(CASE WHEN s.statement_txn_id IS NOT NULL THEN 1 ELSE 0 END) AS matched,
  CAST(SUM(CASE WHEN s.statement_txn_id IS NOT NULL THEN 1 ELSE 0 END) AS DOUBLE)
    / NULLIF(COUNT(a.gateway_txn_id), 0) * 100 AS coverage_pct
FROM v_all_gateway_actions a
LEFT JOIN v_all_statement_rows s
  ON a.gateway = s.gateway AND a.gateway_txn_id = s.gateway_txn_id
GROUP BY 1, 2;
-- v_commissions_monthly: powers commissions widget.
CREATE OR REPLACE VIEW v_commissions_monthly AS
SELECT
  b.BONUS_PERIOD AS month,
  d.COUNTRY_CODE AS market,
  b.DST_ID,
  SUM(b.AMOUNT_USD) AS total_usd,
  COUNT(DISTINCT b.BONUS_TYPE) AS bonus_kinds
FROM dw_bonus b
JOIN dw_dst d ON d.DST_ID = b.DST_ID
WHERE d.DIST_STATUS IN ('A','B','C','H','P','M')
GROUP BY 1, 2, 3;
-- v_rates_matrix: powers rates widget (market x processor).
CREATE OR REPLACE VIEW v_rates_matrix AS
SELECT
  country AS market,
  gateway AS processor,
  COUNT(*) AS orders,
  SUM(CASE WHEN result IN ('approved','settled','captured') THEN 1 ELSE 0 END) AS approved,
  SUM(CASE WHEN result = 'refunded' THEN 1 ELSE 0 END)    AS refunded,
  SUM(CASE WHEN result IN ('chargeback','dispute') THEN 1 ELSE 0 END) AS chargebacks,
  100.0 * SUM(CASE WHEN result IN ('approved','settled','captured') THEN 1 ELSE 0 END)
    / NULLIF(COUNT(*), 0) AS processing_pct
FROM v_all_gateway_actions
WHERE action_date >= CURRENT_DATE - INTERVAL 30 DAY
GROUP BY 1, 2;
-- v_fraud_signals: powers fraud abuse widget.
CREATE OR REPLACE VIEW v_fraud_signals AS
SELECT
  fs.kind,
  d.COUNTRY_CODE AS market,
  fs.severity,
  COUNT(*)          AS count,
  SUM(fs.amount_usd) AS amount_usd
FROM fraud_signal fs
JOIN dw_dst d ON d.DST_ID = fs.DST_ID
WHERE fs.flagged_at >= CURRENT_DATE - INTERVAL 30 DAY
GROUP BY 1, 2, 3;

5.1 Auth

In production: SSO (Azure AD), JWT with portal_9094_read claim, row-level filters enforced at the API layer. In demo: no auth, static files only.

5.2 Caching

Per-endpoint Redis cache, TTL 15 min, keyed on endpoint + querystring. Manual bust via DELETE /api/v1/cache/:key (admin only).

6. ETL & refresh cadence

All pullers are orchestrated by Windows Task Scheduler jobs (prod-equivalent on server: cron / EventBridge Scheduler). Each puller is a standalone Python CLI, idempotent, writes to a staging partition and then atomically swaps. Three cadence tiers are supported per source:

A single umbrella batch (run_all_etl.bat) chains the tier group in order: pullers → dimension join → view refresh → cache bust. On failure the batch exits non-zero so the scheduler surfaces the miss in the Activity Feed. Each puller also respects the per-gateway data floor — nothing older than the floor is requested.

7. Reconciliation model

For each gateway, monthly statements arrive as PDF or CSV. A parser lands them in v_all_statement_rows. The recon gauge is:

coverage_pct = matched_gateway_rows / total_gateway_rows

In practice coverage sits at 60-65% because some gateway-native rows (wallet fees, FX haircuts) never appear on the merchant statement. Anything below 40% is a WARN, below 20% means the parser is broken.

9. Going live

  1. Add .env with DW and UnityDBStage credentials.
  2. Stand up Flask/FastAPI with the endpoints in section 5.
  3. Change data.js from fetch('data/*.json') to fetch('/api/v1/*').
  4. Nothing else in the UI changes.
The front-end is already aligned with the final shape of the API. The bottleneck for going live is auth + a hostname, not UI work.

10. Notes on demo synthesis demo

The file _generate-fake-data.py produces all nine JSONs from a fixed random.seed(9094). Shape fidelity:

  • Approval rates use a mean-reverting random walk toward each market's base_rate (σ ≈ 0.85/day), clamped to [40, 97].
  • Annual seasonality: sin wave peaking Nov-Dec, dipping Jan-Feb.
  • Gateway splits: NMI 40 / Worldpay 28 / Trust 18 / Nuvei-CA 8 / Braintree 4 / Truemed 2.
  • Chargebacks = 0.25% of approved volume + noise. Refunds = 2.5% + noise.
  • Commissions: 40 earners, 30 enrollers, 20 market payout rollups.
  • Fraud: 89 aggregated signals across 9 kinds, 25 individual cases.
  • Rates: 42 market x processor combos with per-processor base-rate adjustments.

Nothing leaks real figures, but the shape matches what an analyst would expect, so the widgets render naturally.
